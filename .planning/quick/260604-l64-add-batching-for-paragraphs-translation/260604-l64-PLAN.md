# Quick Task Plan: Add batching for paragraphs translation

<current_behavior>
- `src/book_translator/translator/engine.py` translates one translatable `Paragraph` per OpenAI chat completion request via `translate_one()` and `translate_paragraph()`.
- `src/book_translator/translator/chunker.py` only returns a symmetric before/after context window around one flat paragraph index.
- `src/book_translator/translator/prompt.py` asks for plain translated text and wraps a single paragraph in `<source_text>` XML tags.
- CLI `--context-window` currently means number of neighboring paragraphs for single-paragraph requests.
- EPUB assembly happens after translation via `assemble(job_dir=run_dir, target_lang=target_lang)` and is independent of prompt/request shape.
</current_behavior>

<target_behavior>
- Translation requests use structured JSON input and require structured JSON output, mapping each requested paragraph ID to exactly one translated string.
- Translation batches contain one or more translatable paragraphs sized to use approximately 60% of the configured/model context budget. Small budgets may yield one paragraph per batch; larger budgets should include multiple paragraphs.
- Every batched request includes additional context:
  - if batch starts at a page/section start, include available page/section/chapter headers (`Chapter.title` and heading paragraphs at the start of that chapter/section);
  - otherwise include up to 3 preceding paragraphs from the same page/section/chapter, including their existing translations when available.
- Batch responses are applied back to the matching `Paragraph.translation` fields by paragraph ID.
- Progress callback still counts completed translatable paragraphs, not completed batches.
</target_behavior>

<non_goals>
- Do not change EPUB assembly behavior.
- Do not change `--mode` semantics or any assembly algorithm controlled by `--mode`; after translation, the same output JSON shape and assembly flow must continue to be used.
- Do not introduce new external dependencies unless impossible with existing stdlib + OpenAI SDK + Pydantic.
</non_goals>

<implementation_notes>
- Prefer a small internal batch model in `translator/chunker.py`, e.g. `TranslationBatch` with `items: list[Paragraph]` and `context: list[...]`, or simple dict/list contracts if avoiding new Pydantic models.
- Preserve `Paragraph.translation` as `str | None`; structured response parsing should end by assigning plain strings to existing paragraphs.
- Estimate token/context usage with a deterministic character heuristic, e.g. chars / 4, because exact model context windows are not available locally. Add a default `context_window` budget that keeps current CLI compatibility; if keeping `--context-window` as count would be ambiguous, treat it as legacy neighbor cap and add internal constants for model-budget approximation.
- For OpenAI structured mode, call `client.chat.completions.create(..., response_format={"type": "json_schema", ...})` or the compatible SDK shape already accepted by installed `openai>=1.0`. The schema should require an object with `translations` array entries containing `id` and `text`.
- Prompt should instruct model to return only JSON matching schema and preserve batch item order, but code must map by `id`, not by order only.
- If malformed/partial JSON is returned, mark only missing/invalid paragraph IDs as `[TRANSLATION FAILED]`; valid IDs from the same response should still be applied.
</implementation_notes>
